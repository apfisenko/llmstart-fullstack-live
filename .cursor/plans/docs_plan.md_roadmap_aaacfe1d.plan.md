# План: `docs/plan.md` (дорожная карта)

## Исходные опорные документы

- [docs/vision.md](docs/vision.md): мультиклиентская платформа; ядро — **backend**; бот — первый клиент; веб — студент + преподаватель; LLM и данные только через backend; краткосрочно допустим узкий контур при сохранении принципа «логика не только в боте».
- [docs/idea.md](docs/idea.md): MVP по смыслу — диалог в Telegram, навигация по курсу, начальная **фиксация прохождения** в боте; затем перенос во веб.
- [docs/data-model.md](docs/data-model.md): `User`, `Cohort`, `CohortMembership`, `Dialogue`, прогресс/submission — эволюция по этапам без «всё сразу».
- [docs/integrations.md](docs/integrations.md): MVP-критичны Telegram, OpenRouter-совместимый LLM, PostgreSQL; веб как альтернативный канал в roadmap.
- [docs/adr/adr-001-database.md](docs/adr/adr-001-database.md): PostgreSQL как основная СУБД.

Текущее состояние репозитория: реализован **бот** (`bot/`, история в памяти процесса по конвенциям). Целевая раскладка из vision — `backend/`, `frontend/bot`, `frontend/web`; дорожная карта описывает **продуктовые этапы**, без привязки к мелким рефакторингам каталогов.

## Содержание файла (структура по ТЗ)

### Организация работ (2–3 предложения)

Зафиксировать: `plan.md` — **единая сквозная дорожная карта** по итерациям; детализация ведётся в **tasklist** по областям ([шаблон](docs/templates/tasklist.md)); принцип **«один этап — один основной tasklist»** (итерация в первую очередь ведётся через один файл `docs/tasks/tasklist-<область>.md`; при необходимости второй tasklist указать в колонке или в тексте итерации).

### Ключевые особенности плана (2–3 принципа)

Предлагаемые формулировки (краткое пояснение под каждым):

1. **Backend-first / одно ядро** — бизнес-правила и данные централизованы в backend; бот и веб — адаптеры (см. vision §2, §10).
2. **API-first для клиентов** — клиенты ходят в backend по HTTP; контракт стабилизируется до масштабного веб-UI (см. vision §14, integrations).
3. **Поэтапная ценность: бот → веб** — сначала сценарии в Telegram (idea §6), затем те же смыслы в веб-кабинете (idea §2).

### Легенда статусов

Точно как в ТЗ: Planned / In Progress / Done с эмодзи.

### Обзор итераций — таблица

Колонки: Итерация | Название | Цель | Статус | Tasklist.

- Статус для всех строк при первой публикации: **Planned**.
- **Tasklist**: markdown-ссылки на целевые файлы (пока каталога `docs/tasks/` нет — это **намеренные пути**; при старте итерации создаются соответствующие `tasklist-*.md` по [workflow](.cursor/rules/workflow.mdc)).

Пример привязки областей к этапам (логика, не жёсткий backlog):

| Итерация | Основной tasklist |
|----------|-------------------|
| 1 | [docs/tasks/tasklist-backend.md](docs/tasks/tasklist-backend.md) |
| 2 | [docs/tasks/tasklist-bot.md](docs/tasks/tasklist-bot.md) (доработки backend — в том же этапе или ссылкой на backend tasklist как доп.) |

В таблице допустимо указывать **одну** основную ссылку на tasklist; при «смешанном» этапе — в блоке итерации явно: «связь: backend + bot».

### Итерации (6 штук — в диапазоне 4–7)

Формат каждой: заголовок **«Итерация N: …»**, **Цель** (одна строка), при необходимости **Параллельность**, **DoD** (маркированный список проверяемых критериев), **Связь с tasklist**, **Полезный результат**, **Артефакты** (только если есть смысл: например ADR, миграции, контракт API).

Предлагаемая логика этапов (высокий уровень):

1. **Фундамент backend и данные** — PostgreSQL, базовые сущности из data-model (пользователь, поток, участие), health/config, минимальный публичный API; DoD: данные переживают рестарт; ADR-001 соблюдён на уровне проекта.
2. **Тонкий бот + LLM через backend** — сценарий «вопрос по курсу» без дублирования логики в боте; диалоги персистентны; DoD: студент ведёт диалог в TG, ответы идут через backend + LLM.
3. **Прогресс и поток** — фиксация прохождения и связь с потоком (в духе idea MVP); DoD: прогресс виден системе и воспроизводим из API; преподавательский сценарий может быть минимальным на API.
4. **Веб: роль студента** — единый веб-клиент, студент: навигация/прогресс/диалог в границах курса; DoD: паритет ключевых студенческих сценариев с ботом (не обязательно 100% фич с первого дня — но явный инкремент «кабинет студента»).
5. **Веб: роль преподавателя** — обзор потока и активности группы (vision §4); DoD: преподаватель видит согласованный с продуктом срез по потоку.
6. **Эксплуатация и качество** — деплой, CI, секреты, наблюдаемость без утечки текста переписки (vision §15); DoD: воспроизводимый деплой и базовые операционные практики.

**Параллельность (для секций):**

- Итерация 4 (веб-студент) можно **частично** параллелить с завершением итерации 3, **если** зафиксирован контракт backend API для студенческих сценариев (основание: vision «клиенты — адаптеры»).
- Итерация 6 может **перекрываться** с 4–5 по инфраструктурным задачам (CI, наблюдаемость), не блокируя функциональные критерии веб-части.

## Что не включать в документ

- Подробный backlog, подзадачи, оценки.
- Перечень эндпоинтов и стек веб-фреймворка (в vision помечен как уточняемый).
