# Проектирование таблиц (PostgreSQL)

## Ориентиры проекта

- Логическая модель и сущности: `@docs/data-model.md`.
- Выбор СУБД и последствия: `@docs/adr/adr-001-database.md`.
- Структура backend и пути к коду — **единый источник** в `@.cursor/rules/conventions.mdc`:
  - миграции **Alembic** — каталог **`backend/migrations/`** (рядом с `backend/alembic.ini`);
  - ORM-модели и доменные типы — **`backend/app/domain/`**;
  - движок БД, сессия, репозитории — **`backend/app/infrastructure/`** (инъекция через `app/api/deps.py`).
- В старых черновиках планов мог встречаться путь `backend/db/migrations/` — в текущем репозитории он **не используется**; ориентир — `backend/migrations/` и дерево `app/` выше.

## Идентификаторы и ключи

- Суррогатные ключи сущностей — **UUID** (`uuid` в PG), не автоинкремент без явной причины.
- Имена таблиц — множественное число, snake_case; первичный ключ обычно `id`.
- Внешние ключи — явные `REFERENCES ... ON DELETE` по смыслу (чаще `RESTRICT`/`NO ACTION` для домена; `CASCADE` только когда удаление родителя должно убирать детей).

## Типы и столбцы

- Текст фиксированной длины — `text` или `varchar(n)` по реальному ограничению; длинные тексты (сообщения) — `text`.
- Время — `timestamptz` для моментов событий; `date` для календарных границ без времени.
- Перечисления в БД: предпочтительно **PostgreSQL ENUM** (отдельный тип) или `smallint`/строка с CHECK — согласовать с тем, как это отражается в SQLAlchemy и в API (snake_case в JSON — по OpenAPI).
- Редкие гибкие поля без жёсткой схемы — **JSONB** с индексом по необходимости (GIN), не как замена нормальной модели.

## Ограничения и целостность

- Уникальность бизнес-комбинаций — `UNIQUE` (например, одно участие пользователя в потоке).
- Обязательность — `NOT NULL` там, где поле всегда есть после создания строки; для поэтапного заполнения — nullable + валидация на уровне приложения.
- Не дублировать в одной строке то, что уже выводится из FK без нужды (избегать лишней денормализации).

## Индексы

- Индекс под каждый типичный фильтр/сортировку в списках API (FK, `created_at`, составные — по реальным запросам).
- Уникальные индексы вместо отдельной таблицы, когда достаточно одного ограничения.
- Частичные индексы (`WHERE ...`) — когда доменно оправдано (например, только «активные» записи).

## Миграции

- Любое изменение схемы — только через Alembic; ручные правки прод-схемы без миграции не предлагать.
- Ломающие изменения: по возможности многошагово (добавить столбец → бэкфилл → переключить код → убрать старое).
- Имена ревизий и операции читаемые; down-миграции осмысленны, если команда их использует.

## Анти-паттерны

- «Умные» типы без нужды, избыточные JSON-деревья вместо нормальных таблиц.
- Мягкое удаление без политики и индексов под типичные выборки.
- Суррогатные ключи там, где естественный составной ключ — реальный идентификатор сущности (редко; по умолчанию UUID).

## Связь с приложением

- Модели SQLAlchemy отражают таблицы 1:1 по смыслу; имена атрибутов Python — согласованы с проектом, колонки — snake_case.
- Не логировать в приложении содержимое чувствительных полей; это не про схему, но про согласованность с правилами backend.
