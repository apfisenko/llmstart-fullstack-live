# Принципы API (backend как ядро)

## Роль API

- Бизнес-правила и данные — **только в backend**; бот и веб — тонкие клиенты.
- Один контракт для всех каналов: избегать «специальных» полей только для Telegram, если то же не нужно вебу (иначе — явная причина в дизайне).

## REST-основы

- Ресурсы — **существительные во множественном числе**: `/flows`, `/submissions`.
- Глаголы — через HTTP: `GET` чтение, `POST` создание, `PUT`/`PATCH` обновление, `DELETE` удаление где уместно.
- Идемпотентность: `PUT`/`DELETE` предсказуемы; `POST` — создание и побочные эффекты.
- Вложенность только при реальной иерархии: `/flows/{id}/participants`, не глубже без нужды.

## Версионирование

- Префикс версии в пути: `/api/v1/...`. Ломающие изменения — новая мажорная версия (`v2`), не «тихие» правки полей в `v1`.

## Ответы и ошибки

- Успех: соответствующий 2xx; тело — стабильная JSON-схема (документировать).
- Ошибки: единый формат, например `{"detail": "...", "code": "VALIDATION_ERROR"}` (конкретные поля — по соглашению проекта). Не отдавать внутренние трейсы клиенту.
- Коды: 400 валидация/битый запрос, 401/403 авторизация, 404 отсутствие ресурса, 409 конфликт, 422 тело не проходит схему (если используется), 5xx только неожиданные сбои сервера.

## Безопасность и контракт

- Чувствительные операции только с аутентификацией; права по ролям — явно в спецификации.
- Пагинация/фильтры для списков: `limit`/`offset` или cursor — зафиксировать и не ломать без версии.

## Документация

- OpenAPI как источник правды для публичных маршрутов; при изменении API — обновлять схему и примеры.
