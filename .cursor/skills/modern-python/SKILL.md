# Современный Python (проект)

## Ориентиры

- Общие правила: `@.cursor/rules/conventions.mdc` (KISS, без лишних фреймворков).
- Стек: Python **3.9+**, **uv**, **pyproject.toml**, **ruff**, **pytest**.

## Инструменты

- Зависимости и окружение — **uv**; не добавлять pip-tools/poetry без необходимости.
- Линт и формат — **ruff** (настройки в `pyproject.toml` проекта).
- Правила типов и линта — из `[tool.ruff]` в `pyproject.toml` пакета; не ослаблять локально без причины.

## Типизация

- Аннотации у публичных функций и значимых внутренних; возвращаемый тип явно, если не очевиден.
- Для совместимости с 3.9: `list[str]`, `dict[str, Any]` допустимы с `from __future__ import annotations` в модуле или классический `typing.List` — как уже принято в соседних файлах.
- Избегать голого `Any`; для опциональных полей — `T | None` или `Optional[T]`.
- `TypedDict` / `@dataclass` — когда структура данных реально нужна; не плодить датаклассы на каждый словарь.

## Async

- В async-эндпоинтах и сервисах — только неблокирующие вызовы: async-драйверы БД, `httpx.AsyncClient`, не `time.sleep` / синхронный IO в горячем пути.
- Явные `async def` для операций с БД и HTTP; не смешивать в одной функции тяжёлый CPU и async без нужды (вынести или `asyncio.to_thread` только осознанно).

## Pydantic и конфиг

- **pydantic-settings**: обязательные поля без «тихих» дефолтов — падаем при старте с понятной ошибкой (как в `Config` проекта).
- Pydantic v2: `model_config`, `Field(...)`; валидаторы и сериализаторы — по документации v2, не v1.

## Стиль кода

- Имена говорящие; код без поясняющих комментариев к очевидному — предпочтение ясным именам и маленьким функциям.
- **pathlib** для путей; f-строки для форматирования; контекстные менеджеры для ресурсов.
- Исключения: конкретные типы; не глотать `Exception` без переброса или лога; не использовать исключения для обычного потока управления.

## Импорты и модули

- Порядок импортов и группы — как в остальном пакете (stdlib → third party → local); избегать циклов через локальные импорты только при необходимости.
- Один класс на файл — для `bot/` по соглашению; в `backend/app/` — как уже устроено в соседних модулях.

## Тесты

- **pytest**, асинхронные тесты с принятыми в проекте маркерами/фикстурами; HTTP — `httpx` + `ASGITransport` для FastAPI.
- Подробности: skill `python-testing-patterns`.

## Анти-паттерны

- «Универсальные» базовые классы, слои абстракций «на вырост», динамические магии без нужды.
- Глобальное изменяемое состояние вместо DI / параметров.
- Зависимости вне `pyproject.toml` или версии «как получится».
