# Паттерны тестирования (Python)

## Инструменты

- **pytest** — по умолчанию; один тест = одна проверяемая идея.
- Асинхронный код: **pytest-asyncio** (`async def` тесты, маркер/режим по документации плагина).
- FastAPI: **httpx** `AsyncClient` + `transport=ASGITransport(app=...)` или `TestClient` для синхронных сценариев.

## Структура

- Тесты рядом с кодом (`tests/`) или зеркало дерева — как принято в репо; **не** дублировать бизнес-логику в тестах — вызывать те же сервисы/функции, что и прод.
- `conftest.py`: общие фикстуры (приложение, клиент, временные данные).

## Фикстуры

- Область (`scope`): `function` по умолчанию; `session` для дорогого одноразового setup.
- Фикстуры возвращают готовое состояние; cleanup — через `yield` или финализаторы.

## Хорошие практики

- Имена тестов: `test_<что>_<ожидание>` или `test_<метод>_<условие>`.
- **Параметризация** (`@pytest.mark.parametrize`) для таблиц входов вместо копипаста.
- Моки внешних HTTP/LLM: стабильные ответы, не хрупкая привязка к полному телу если достаточно статуса/ключевых полей.
- Не полагаться на порядок выполнения тестов; каждый тест независим.

## Что проверять в API-тестах

- Статус и тело для happy path и типовых ошибок (400/401/404).
- После мутаций — GET или прямой запрос к слою данных, если доступен в тестах.

## Запуск

- Локально: `pytest` или `uv run pytest`; при необходимости `-q`, `-k`, `--tb=short`.
