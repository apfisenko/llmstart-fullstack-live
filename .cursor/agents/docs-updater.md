# Subagent: docs-updater

Специализированный сценарий для агента: **привести документацию в соответствие с фактическим API**, не дублируя OpenAPI целиком в prose.

## Триггеры (когда запускать)

- Изменены файлы под `backend/app/api/v1/` (роуты, зависимости, схемы).
- Изменён `docs/api/openapi-v1.yaml` или договорились считать источником `GET /openapi.json`.
- Добавлены/переименованы публичные маршруты, query/body/path-параметры, коды ответов, семантика ролей (401/403), гостевой ассистент.
- Пользователь просит обновить контракт документации после рефакторинга API.

Не запускать **только** из-за внутренних изменений сервисов без impact на контракт HTTP.

## Иерархия истин (при расхождении)

1. Запущенный backend: **`GET /openapi.json`** (и при необходимости проверка маршрутов в коде).
2. Репозиторий: **`docs/api/openapi-v1.yaml`**, если live OpenAPI недоступен.
3. Код: `backend/app/api/v1/**/*.py`, `backend/app/main.py` (служебные `/health`, `/health/db`).

Контракт для людей и клиентов — **`docs/tech/api-contracts.md`**; он должен совпадать по смыслу с приоритетным источником выше.

## Действия

1. **Собрать факт**: список метод/путь, назначение в одну строку, заметные query/path поля, изменения в авторизации и ошибках.
2. **Обновить `docs/tech/api-contracts.md`**:
   - таблица «Эндпоинты» и при необходимости блок «Веб: сценарии по экранам», «Сопоставление с доменом», «Идентификация», «Ошибки»;
   - сохранить стиль: краткие описания, отсылка к OpenAPI для полных тел JSON;
   - если менялись только детали тел — достаточно указать это в тексте и убедиться, что YAML/OpenAPI отражает детали.
3. **При необходимости — `docs/onboarding.md`**:
   - разделы со сценариями и примерами вызовов (например dev-session), пути к `backend/app/api/v1/`, любые упоминания конкретных URL или шагов, которые перестали быть верными;
   - не дублировать полный список эндпоинтов — ссылка на `docs/tech/api-contracts.md` предпочтительнее.
4. **Синхронизация OpenAPI**: если менялся только markdown без правок кода — проверить, что YAML не противоречит backend; если менялся код — обновить **`docs/api/openapi-v1.yaml`** в той же задаче/коммите, если это принято в команде (иначе явно отметить техдолг).

## Ограничения

- Не логировать и не включать в доки тексты сообщений пользователей/LLM (см. conventions).
- Не раздувать `api-contracts.md` полными JSON-примерами — только то, что нужно для ориентации клиентов.
- Не править `docs/tasks/**` и vision/plan без отдельного запроса.

## Проверка готовности

- В `api-contracts.md` нет ссылок на несуществующие пути относительно актуального списка эндпоинтов.
- Примеры в onboarding (curl, username, шаги) соответствуют текущему контракту или явно помечены как временные.
