# Требования к веб-клиенту (frontend)

Единая сводка требований к веб-приложению платформы. Детали контракта HTTP — в [`api-contracts.md`](api-contracts.md) и [`docs/api/openapi-v1.yaml`](../api/openapi-v1.yaml); продуктовые границы — в [`docs/vision.md`](../vision.md); декомпозиция работ — в [`docs/tasks/tasklist-frontend.md`](../tasks/tasklist-frontend.md).

---

## 1. Роль веб-клиента в системе

- Веб — **клиент ядра (backend)**: только HTTP **`/api/v1/`**, как у Telegram-бота ([`vision.md`](../vision.md) §1–2).
- **Запрещено:** дублирование бизнес-логики сопровождения потока, прямой вызов LLM из браузера, прямой доступ к БД.
- Разграничение **студент / преподаватель** — на уровне UI, маршрутов и передаваемых в API идентификаторов (`membership_id`, `viewer_membership_id`, роли из ответов backend).

## 2. Стек и артефакты

| Область | Требование |
|---------|------------|
| Фреймворк | **Next.js** (App Router) |
| Язык | **TypeScript** |
| UI | **shadcn/ui** (Radix + Tailwind) |
| Стили | **Tailwind CSS** |

**Рекомендуемые skills:** `find-skills`, `shadcn-ui`, `vercel-react-best-practice`, `nextjs-app-router-patterns`, `api-design-principles`, `sharp-edges` (см. [`.cursor/skills/`](../../.cursor/skills/)).

## 5. Вход и сессия (MVP)

- **Без OAuth и без сложной авторизации:** пользователь вводит **Telegram username** (`@` опционален).
- После успешного **`POST /api/v1/auth/dev-session`** клиент сохраняет контекст (**`membership_id`**, **`cohort_id`**, роль, при необходимости **`web_dialogue_id`**) — например `localStorage`; далее те же запросы с общим **`Authorization: Bearer`** (как у бота). Отдельного JWT для пользователя нет.
- На клиенте: отображение информации о пользователе и **кнопка выхода** (сброс сохранённого контекста).
- Резолвинг username → пользователь на backend опирается на колонку **`users.telegram_username`** (см. [`data-model.md`](../data-model.md) §`User`); не использовать «только сид-маппинг» без поля в БД как источник контракта.
- **Не передавать** `telegram_username` в query/path `GET` (логи прокси).

Подробнее: [`docs/integrations.md`](../integrations.md), [`api-contracts.md`](api-contracts.md) §«Идентификация», план задачи 01.

## 6. Зоны интерфейса A–D и экраны

Нумерация зон согласована с задачей 01 и tasklist.

### 6.1. Зона A — панель преподавателя

Один экран — сразу понятно состояние потока ([`ui-requirements.md`](../ui/ui-requirements.md) экран 1).

| Блок | Требование |
|------|------------|
| KPI (4) | Студенты в потоке; домашки сданы (агрегат по группе); средний прогресс; вопросы к ассистенту за период. У каждого KPI — **сравнение с предыдущей календарной неделей** (дельта / динамика). |
| График | Линейный график: число вопросов к ассистенту **по дням** за **14 дней**; допускаются пресеты периода через API (`activity_days` и т.п.). |
| Лента вопросов | Кто, когда, текст вопроса и ответ ассистента; **пагинация**; при поддержке API — **поиск** по тексту вопроса (`q`). |
| Лента сдач | Кто, что сдал, когда; по клику — **текст отчёта** и **ссылки на работы**. |
| Матрица | Строки — студенты, столбцы — уроки (чекпоинты); ячейка — факт сдачи; последний столбец — счёт «из скольких»; сортировка по счёту; **при наведении** — дата сдачи (`updated_at`). |

**Данные:** предпочтительно один запрос **`GET /api/v1/cohorts/{cohort_id}/teacher-dashboard`** (см. план задачи 01); при необходимости дополнительно `GET .../summary`.

**Состояния UI:** loading / empty / error для всех блоков с данными с сервера.

### 6.2. Зона B — лидерборд

Публично-ориентированный мотивационный экран ([`ui-requirements.md`](../ui/ui-requirements.md) экран 2).

| Элемент | Требование |
|---------|------------|
| Переключатель | Два вида: **таблица** и **scatter plot** («координаты»). |
| Таблица | Место в рейтинге; прогресс-бар; счёт (например N из M); **иконки по каждому уроку** (сдано / нет); **топ-3** — визуально золото / серебро / бронза. |
| Scatter | Точка = студент; горизонталь — продвижение по **урокам** (не ДЗ); вертикаль — **домашки** (`is_homework`); **tooltip** — имя и цифры. |

**Данные:** `GET /api/v1/cohorts/{cohort_id}/leaderboard`; заголовки колонок / признак ДЗ — `GET .../progress-checkpoints` с полем **`is_homework`**. Один ответ лидерборда для обоих видов — клиент не пересчитывает рейтинг, если API уже отдаёт порядок.

### 6.3. Зона C — личный кабинет студента

[`ui-requirements.md`](../ui/ui-requirements.md) экран 3.

| Элемент | Требование |
|---------|------------|
| Шапка | Приветствие по имени; кнопка перехода в чат с ассистентом. |
| Сводка | Сколько уроков сдано из общего числа; какой **следующий** урок. |
| Список уроков | Сданные — с датой (зелёный акцент по смыслу UI-дока); текущий — кнопка **«Сдать задание»**; будущие — **заблокированы** (правило на клиенте по `sort_order` и статусам с сервера). |
| Форма сдачи | Текстовый отчёт (**`comment`**) и **ссылки на работы** — отдельное поле / массив **`submission_links`** в `PUT .../progress-records/{checkpoint_id}`; не смешивать ссылки только в `comment`. |

**Данные:** `GET /api/v1/cohorts/{cohort_id}/memberships/{membership_id}/progress-overview`.

### 6.4. Зона D — чат с ассистентом

Глобальный чат ([`ui-requirements.md`](../ui/ui-requirements.md) §«Чат с AI-ассистентом»).

| Элемент | Требование |
|---------|------------|
| Доступность | **FAB** на всех экранах; панель **поверх** контента; закрытие — возврат к тому же маршруту. |
| Содержимое панели | История сверху (прокрутка вверх), ввод снизу; отправка и отображение ответа (задержка допустима; streaming — по возможностям API и согласованию). |
| API | `GET /api/v1/dialogues/{dialogue_id}/turns`; `POST /api/v1/cohorts/{cohort_id}/dialogues/messages` с **`channel: web`**; `POST /api/v1/dialogues/{dialogue_id}/reset`. |
| Конфиденциальность | Не логировать и не выводить в UI лишние персональные данные; соблюдать правила проекта (см. [`vision.md`](../vision.md) §15 — без утечки переписки в ошибках и логах клиента в проде). |

**Страница «Чат»** (основная область, пункт меню): полноэкранная история и тот же диалоговый контур; состояние **`dialogue_id`** согласовано с плавающим виджетом (один диалог на пользователя/канал `web` или явное разделение — фиксировать в плане задачи 07). Индикатор отправки, ошибки API, удобная прокрутка.

## 8. Качество и проверки (MVP)

- Линт и сборка: `pnpm lint` / `make frontend-lint`, production build без ошибок.
- Ревью по **`vercel-react-best-practice`** и **`nextjs-app-router-patterns`** (RSC/Client, данные, кэш); критические проблемы устранить (задача 08).
- Сквозная проверка: **вход → дашборд / лидерборд / чат** на mock-данных backend; убедиться в отсутствии прямых вызовов LLM из браузера.

---

## 9. Вне объёма MVP (итерации / задачи 09–10)

Не входят в обязательные требования первого веб-MVP, пока не запланированы отдельно:

- **Голосовой режим чата** (веб + бот) — tasklist [задача 09](../tasks/tasklist-frontend.md): **отменена** (⛔), реализация не планируется из-за технического усложнения; при возврате к теме — заново оценить roadmap и ADR.
- **Text-to-SQL** и ответы по данным БД (веб + бот через backend) — tasklist [задача 10](../tasks/tasklist-frontend.md): **не выполняется** (⛔) после оценки сложности; зафиксировано в [`technical-debt.md`](technical-debt.md) §«Отложенное развитие»; при возврате к теме — заново оценить roadmap и ADR.
