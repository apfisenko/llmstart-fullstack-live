# Пользовательские сценарии и требования к данным

Документ описывает, **что делают** студент и преподаватель в рамках учебного этапа курса, и **какие данные** для этого нужны системе — без стека, SQL и деталей API. Термины согласованы с [`data-model.md`](../data-model.md) и [`api-contracts.md`](api-contracts.md).

Сквозные сценарии: одинаковый смысл для **Telegram-бота** и **веб-клиента**; различается только интерфейс.

---

## Допущения учебного этапа

| Допущение | Смысл |
|-----------|--------|
| **Модуль == урок** | В программе потока для отображения и прогресса используется **один уровень**: «урок» и «модуль» не разводятся; этапы прогресса (`ProgressCheckpoint`) трактуются как шаги этого единого уровня (в т.ч. домашнее задание как отдельный этап, если так задано в потоке). |
| **История диалога для аналитики == пара «вопрос + ответ»** | Минимальная единица переписки с ассистентом для отчётов преподавателя: **один вопрос пользователя** и **соответствующий ответ ассистента** (один «ход»). Технически в хранилище могут быть отдельные сообщения; для сценариев «активность» и «статистика» считаем такие пары. |
| **Гостевой режим ассистента** | Диалоги без привязки к потоку и участию (гостевой сценарий в API) **не входят** в аналитику потока и в сценарии ниже. |

## Видимость и приватность (кратко)

| Данные | Студент | Преподаватель |
|--------|---------|----------------|
| Собственный прогресс | да | нет (кроме общей сводки по группе) |
| Лидерборд по потоку | обычно **агрегаты** (место, счётчики выполненных этапов, отображаемое имя) — **без** текста чужих диалогов | может совпадать с тем, что видит студент, плюс детальные обзоры по группе |
| Тексты вопросов студентов в диалогах | только **свои** | **все** студенты потока (в рамках политики курса) |
| Списки участников и сводки по прогрессу группы | нет | да |

## Студент

### Сценарий: текущее состояние своего прогресса

**Описание.** Студент открывает экран прогресса по **своему** учебному потоку и видит: какие этапы (уроки/ДЗ в смысле допущения выше) **не начаты**, **в работе**, **выполнены** или **пропущены**; понимает, **что дальше** по программе потока.

**Нужные данные**

- **Пользователь** (`User`) и его **участие в потоке** (`CohortMembership`) с ролью «студент» и идентификатором **потока** (`Cohort`).
- Каталог **этапов прогресса** потока (`ProgressCheckpoint`): порядок, заголовки, привязка к потоку.
- **Фиксации прогресса** (`ProgressRecord`) по этому участию: для каждого этапа — статус и время обновления (и при необходимости краткий комментарий).

**Запросы к данным (смысл)**

- Получить участие студента в выбранном потоке.
- Получить список этапов прогресса для потока (упорядоченный).
- Получить все отметки прогресса студента по этим этапам.

---

### Сценарий: лидерборд по выполнению домашних заданий в потоке

**Описание.** Студент видит **сравнение с другими участниками-потока** по мере выполнения работ: например **рейтинг** или таблицу с **отображаемыми именами** и **числом выполненных этапов**, отнесённых к домашним заданиям (в учебном этапе это те же `ProgressCheckpoint`, помеченные в программе потока как ДЗ, или отдельный поднабор этапов — правило задаётся при наполнении потока, не в этом документе).

Для **минимально достаточной** версии учебного этапа достаточно агрегата: **количество этапов со статусом «выполнено»** (и при необходимости «в работе») среди этапов, отнесённых к ДЗ, **без весов и дедлайнов**. Веса, штрафы за срок и т.п. — расширение на следующие этапы продукта.

**Нужные данные**

- **Поток** (`Cohort`).
- **Участия** в потоке со ролью «студент» и связанные **пользователи** (для отображаемого имени).
- **Этапы** (`ProgressCheckpoint`), отнесённые к категории «ДЗ» (или явный список таких этапов для потока).
- **Фиксации прогресса** (`ProgressRecord`) по каждому студенческому участию и этим этапам.

**Запросы к данным (смысл)**

- Для потока: список студенческих участий с публичным для лидерборда именем.
- Для каждого такого участия: посчитать число завершённых (и при необходимости «в работе») записей прогресса по этапам-ДЗ.
- Отсортировать и отдать агрегированный срез **без** содержимого чужих диалогов.

**Пометка этапа как «ДЗ».** В [`data-model.md`](../data-model.md) зафиксированы логическое поле и колонка БД `is_homework` у этапа прогресса (`ProgressCheckpoint` / `progress_checkpoints`) для лидерборда и фильтров по домашним заданиям. В OpenAPI v1 поле в `ProgressCheckpointItem` при необходимости добавляется отдельной задачей.

## Преподаватель

### Сценарий: участники потока

**Описание.** Преподаватель открывает обзор **участников** выбранного потока: кто состоит в группе, **роль** в потоке (студент / преподаватель), **статус участия** (активен и т.д.), отображаемое имя.

**Нужные данные**

- **Поток** (`Cohort`).
- Все **участия** (`CohortMembership`) в этом потоке с ролями и статусами.
- Связанные **пользователи** (`User`) для имён и идентификаторов.

**Запросы к данным (смысл)**

- Список участий по потоку с подгрузкой профиля пользователя.

---

### Сценарий: прогресс группы

**Описание.** Преподаватель видит **сводку по группе**: по каждому этапу программы (урок/ДЗ в смысле допущений) — сколько студентов в каком статусе (не начато / в работе / выполнено / пропущено) или эквивалентная матрица «студент × этап».

**Нужные данные**

- **Поток**, **этапы** (`ProgressCheckpoint`) потока.
- Все **студенческие участия** в потоке.
- **Фиксации прогресса** (`ProgressRecord`) по этим участиям и этапам.

**Запросы к данным (смысл)**

- Этапы потока (упорядоченно).
- Для каждого студента потока — все `ProgressRecord` по этим этапам **или** агрегированные счётчики по этапам для всей группы.

---

### Сценарий: активность за период по диалогам (вопросы студентов)

**Описание.** Преподаватель задаёт **интервал дат** (или «неделя / месяц» в UI) и видит **активность** студентов в части обращений к ассистенту: кто задавал вопросы, **когда** был задан вопрос (в рамках пары «вопрос+ответ»), к какому **диалогу** это относится. «Активность» здесь = **факт и время пользовательского вопроса** (начало пары), без обязательного показа полного текста в сводном списке (детали — по раскрытию записи).

**Нужные данные**

- **Диалоги** (`Dialogue`), привязанные к **участию** в потоке и потоку.
- **Сообщения** (`DialogueMessage`) с ролью «пользователь» и временем создания; для связки с «парой» — соседний ответ ассистента (роль «ассистент») в том же диалоге (по времени/порядку).

**Запросы к данным (смысл)**

- Диалоги по потоку (через участия) с фильтром по интервалу по дате **пользовательских** сообщений.
- Подсчёт числа вопросов по студенту за период (как минимум).

---

### Сценарий: статистика и агрегаты по истории диалогов

**Описание.** Преподаватель получает **отчёты-агрегаты** за выбранный период, например:

- общее **число вопросов** (пар «вопрос+ответ») по потоку;
- распределение вопросов **по студентам** (кто сколько спрашивал);
- привязка к **урокам/этапам**, если в продукте появится явная метка контекста вопроса; **на учебном этапе** без такой метки достаточно агрегатов по пользователю и по времени (столбцы «по урокам» — задел на следующую итерацию, если контекст не хранится).

**Нужные данные**

- Те же, что в сценарии активности: `Dialogue`, `DialogueMessage` (пользователь/ассистент), связь с `CohortMembership` и `Cohort`.

**Запросы к данным (смысл)**

- За период: COUNT пар «вопрос+ответ» по потоку.
- GROUP BY студент (участие): COUNT вопросов.
- Опционально (если в модели появится связь вопроса с `ProgressCheckpoint` или уроком): GROUP BY этапу — **иначе зафиксировать пробел** в продукте.

**Пробел.** Явная связь «вопрос ассистенту ↔ урок потока» в текущей модели не обязательна; сценарий «статистика по урокам» потребует либо тегирования сообщений, либо отдельного поля контекста — на усмотрение задачи 02.
