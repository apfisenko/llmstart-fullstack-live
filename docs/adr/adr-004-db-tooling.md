# ADR-004: практики и соглашения по работе с БД в репозитории

## Статус

Принято

**Дата принятия:** 2026-04-12

## Контекст

[ADR-001](adr-001-database.md) фиксирует СУБД (PostgreSQL). [ADR-002](adr-002-backend-stack.md) фиксирует стек backend: **SQLAlchemy 2.x (async)**, драйвер **asyncpg**, **Alembic**, модели в `app/domain/`, конфигурация через **pydantic-settings**.

Нужно отдельно зафиксировать **как в этом репозитории** применяются миграции и ORM: синхронный Alembic поверх async-URL, именование таблиц и миграций, расположение репозиториев и сессии — чтобы не дублировать длинные инструкции в ADR-002 и иметь одну ссылку для задач по данным.

## Решение

1. **Стек доступа к данным** — без изменений относительно ADR-002: приложение использует **async** SQLAlchemy и **asyncpg** для runtime; схема описана в [`data-model.md`](../data-model.md).

2. **Alembic в репозитории** — ревизии в `backend/migrations/`, конфигурация `backend/alembic.ini` (`script_location = migrations`). `env.py` подключается к БД через **синхронный** SQLAlchemy `create_engine`. Переменная окружения **`DATABASE_URL` обязательна** для команд Alembic. Если URL содержит `+asyncpg`, перед подключением выполняется замена на **`+psycopg2`** (драйвер из dev-зависимостей backend для миграций). Это технический компромисс: одна и та же переменная для приложения и для Alembic, без отдельного «sync URL» в проде для типичного сценария.

3. **Соглашения по схеме** (согласованы с [`data-model.md`](../data-model.md) и skill postgresql-table-design):
   - имена таблиц — **множественное число**, **snake_case** (`users`, `cohort_memberships`, `dialogue_turns`);
   - суррогатные ключи — **UUID** там, где так зафиксировано в модели;
   - моменты времени — **`timestamptz`**;
   - перечисления домена — типы PostgreSQL **ENUM** (изменение набора значений только через миграцию).

4. **Репозитории и сессия** — классы доступа к данным размещаются в `backend/app/infrastructure/`; ORM-модели — в `backend/app/domain/`. Сессия БД передаётся в обработчики через **FastAPI Depends** (`deps.py`), без лишних слоёв абстракции сверх соглашений проекта.

Пошаговые команды и структура каталогов — в [`docs/tech/db-tooling-guide.md`](../tech/db-tooling-guide.md).

## Управление миграциями

- **Инструмент** — только **Alembic**; правки схемы на стендах без ревизии в репозитории не используются как норма (см. skill `postgresql-table-design`).
- **Расположение** — `backend/alembic.ini`, каталог ревизий `backend/migrations/`, файлы версий в `backend/migrations/versions/`. Метаданные для autogenerate/согласованности — `Base.metadata` из `app.domain` (см. `env.py`).
- **Подключение** — в `env.py` используется **синхронный** SQLAlchemy `create_engine` (не async-сессия Alembic). Переменная **`DATABASE_URL` обязательна**; значение должно указывать на ту же БД, что и у приложения в данном окружении.
- **Подмена драйвера** — если в URL есть `+asyncpg`, для миграций строка заменяется на `+psycopg2`, чтобы Alembic открывал соединение через **psycopg2** (`psycopg2-binary` в optional-dependencies dev). Без dev-зависимостей и без Postgres URL команды миграций могут быть недоступны или падать — см. [`db-tooling-guide.md`](../tech/db-tooling-guide.md).
- **Применение** — из корня репозитория: `make migrate-backend`; из `backend/`: `uv run alembic upgrade head` (после `uv sync --extra dev` при работе с Postgres).
- **Именование ревизий** — стабильный идентификатор в поле `revision` и читаемое имя файла; цепочка `down_revision` не разрывается без осознанного ветвления.
- **Согласование с кодом** — после изменения ORM-моделей новая ревизия добавляется в том же изменении/итерации, что и код, читающий новую схему (или многошаговая миграция по правилам skill).

## Рассмотренные альтернативы

Выбор **ORM / отказ от ORM / Tortoise / синхронный SQLAlchemy** для приложения разобран в ADR-002 — здесь не повторяется.

**Отдельный async-режим Alembic** — возможен, но усложняет `env.py` и поддержку; текущий репозиторий сознательно использует **синхронный** движок Alembic с `psycopg2` для миграций при том, что runtime остаётся async + asyncpg.

**Библиотека `databases` (encode)** — слой поверх драйвера без полноценной ORM под нашу реляционную модель; не выбран как основа ядра (см. также сырой SQL / asyncpg в ADR-002).

## Последствия

**Положительные**

- Одна переменная `DATABASE_URL` для локальной разработки и CI; понятное правило подмены драйвера только в контексте Alembic.
- Именование и типы согласованы с `data-model.md` — меньше расхождений между документом и миграциями.

**Отрицательные / риски**

- Для `alembic upgrade` нужны dev-зависимости с **psycopg2** (`uv sync --extra dev` в `backend/`).
- Подмена `+asyncpg` → `+psycopg2` не покрывает экзотические варианты URL; при смене драйвера нужно обновить `env.py` и справку.

## Резюме

Итоговый стек технологий доступа к данным и миграций: в колонке «Версия» — нижние границы из [`backend/pyproject.toml`](../../backend/pyproject.toml); закреплённые версии — в [`backend/uv.lock`](../../backend/uv.lock).

| Уровень | Инструмент | Версия |
|---------|------------|--------|
| Платформа | Python | >=3.9 |
| СУБД | PostgreSQL | не фиксируется в репозитории (см. [ADR-001](adr-001-database.md)) |
| ORM (async API) | SQLAlchemy | >=2.0.36 |
| Драйвер БД (runtime, Postgres) | asyncpg | >=0.30 |
| Миграции схемы | Alembic | >=1.14 |
| Драйвер БД (миграции, синхронный Postgres) | psycopg2-binary | >=2.9; optional `dev` |
| Конфигурация URL и окружения | pydantic-settings | >=2.6 |

Подробности по командам, `DATABASE_URL` и каталогам — в разделе **Управление миграциями** выше и в [`db-tooling-guide.md`](../tech/db-tooling-guide.md).
