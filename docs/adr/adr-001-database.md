# ADR-001: основная СУБД проекта — PostgreSQL

## Статус

Принято

**Дата принятия:** 2026-04-05

## Контекст

Система сопровождения учебного потока — **мультиклиентская** платформа (бот, веб), с **единым backend** и **реляционной предметной моделью**: пользователи, потоки, участия, диалоги, прогресс (см. [`data-model.md`](../data-model.md)). Нужно зафиксировать **первое архитектурное решение** по персистентности: одна основная СУБД для OLTP-домена на MVP и путь развития без смены парадигмы при появлении нескольких сервисов.

Требования к выбору:

- явные **связи и ограничения целостности** (FK, уникальность, транзакции);
- **конкурентная запись** от нескольких воркеров backend и в перспективе — от нескольких сервисов;
- возможность **эволюции схемы** (миграции) и предсказуемая эксплуатация (бэкапы, мониторинг);
- опционально — **полуструктурированные поля** без немедленной жёсткой схемы под каждый эксперимент;
- предсказуемый **хостинг** (в т.ч. managed-облако).

## Решение

Принять **PostgreSQL** как **основную и целевую СУБД** для доменных данных сопровождения потока на **MVP и дальнейшее развитие**.

Кратко о применении:

- один или несколько инстансов/схем по мере дробления сервисов — без смены СУБД как дефолта для этого домена;
- типы **UUID**, **JSONB** при необходимости, стандартные механизмы транзакций и изоляции;
- чтение под нагрузкой — через **реплики** при росте; тяжёлая аналитика и полнотекст — при необходимости **дополняются** отдельными хранилищами, а не заменой Postgres для ядра учебного контура.

## Рассмотренные альтернативы

### 1. SQLite

**Плюсы:** простота локальной разработки, нулевая администрирование на одном файле, достаточно для прототипа одного процесса.

**Минусы:** слабее для **нескольких пишущих процессов** и типичного сетевого доступа «много backend-воркеров → одна БД»; ограничения для **нескольких сервисов** с раздельным жизненным циклом; выбор как «целевой» СУБД создал бы риск миграции на реляционный сервер позже.

**Вывод:** допустим как **временный** слой для экспериментов или embedded-сценариев, но **не** как зафиксированное целевое решение для продукта с описанной архитектурой.

### 2. MySQL / MariaDB

**Плюсы:** широкая распространённость, привычность, хорошая поддержка в PaaS.

**Минусы:** для команды, ориентированной на **Postgres-типы и JSONB**, выше трение; различия в поведении индексов и DDL; часть возможностей (расширения, подход к JSON) отличается — без выигрыша для текущей модели.

**Вывод:** возможный технический компромисс в организации с жёсткой стандартизацией на MySQL; для данного проекта **нет преимущества** над PostgreSQL.

### 3. Документоориентированная БД (MongoDB и аналоги)

**Плюсы:** гибкость схемы, удобство вложенных документов.

**Минусы:** доменная модель **изначально реляционная** (потоки, участия, FK); перенос целостности в приложение усложняет backend; для транзакций между сущностями и отчётов по потоку реляционная модель естественнее.

**Вывод:** не выбрана как основная OLTP-СУБД для ядра; при необходимости **дополнительное** хранилище рассматривается точечно (не в рамках ADR-001).

### 4. «Только облачные» проприетарные СУБД под одного вендора

**Плюсы:** глубокая интеграция с экосистемой провайдера.

**Минусы:** привязка к вендору и сложнее перенос; для учебного продукта и небольшой команды **PostgreSQL как открытый стандарт** предпочтительнее как базовый выбор.

**Вывод:** не исключает **managed PostgreSQL** у выбранного облака; решение — про **движок**, а не про конкретный SaaS с уникальным API.

## Последствия

**Положительные**

- Единый язык с [`data-model.md`](../data-model.md) и с типичным стеком backend (миграции, ORM/SQL).
- Готовность к **горизонтальному чтению** и к **нескольким схемам/БД** при росте числа сервисов.
- Меньше споров о «второй базе» на старте: Postgres покрывает OLTP и умеренную гибкость через JSONB.

**Отрицательные / риски**

- Нужны **процедуры миграций** и дисциплина версий схемы.
- Администрирование инстанса (даже managed) — ответственность команды; проще, чем у кластеров NoSQL, но не «нулевая».
