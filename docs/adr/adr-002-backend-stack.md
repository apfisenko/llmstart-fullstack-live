# ADR-002: HTTP-слой и доступ к данным в backend

## Статус

Принято

**Дата принятия:** 2026-04-06

## Контекст

Backend является ядром платформы (см. [`vision.md`](../vision.md) §2): HTTP API для Telegram-бота и веб-клиента, интеграция с LLM, работа с реляционной моделью данных (см. [`data-model.md`](../data-model.md)). СУБД зафиксирована ADR-001 — PostgreSQL.

Нужно зафиксировать:
- HTTP-фреймворк и ASGI-сервер;
- инструмент доступа к БД (ORM / драйвер) и подход к миграциям;
- конфигурационный инструмент.

Требования:

- **async-first**: несколько воркеров, одновременные HTTP-запросы и запросы к БД/LLM;
- **явная схема API** (OpenAPI/Pydantic) — основа контракта для клиентов и задача 02;
- **безопасный конфиг** — fail fast при отсутствии обязательных переменных;
- **тестируемость** — HTTP-слой и сервисы тестируются независимо;
- **минимум ненужных абстракций** (KISS/YAGNI, принципы проекта).

## Решение

### HTTP-фреймворк: FastAPI + uvicorn

**FastAPI** с ASGI-сервером **uvicorn**.

- Генерирует OpenAPI-схему из аннотаций типов — контракт для задачи 02 без дублирования.
- Pydantic-модели для запросов и ответов — единый язык с домениными схемами.
- Встроенный `lifespan` для управления ресурсами при старте/остановке.
- `TestClient` / `httpx.AsyncClient` для интеграционных тестов.

### Доступ к БД: SQLAlchemy 2.x (async) + asyncpg

**SQLAlchemy 2.0+ (async API)** с драйвером **asyncpg**.

- Async-сессия (`AsyncSession`) совместима с FastAPI Depends и uvicorn workers.
- ORM-модели в `domain/` — единый источник правды для схемы и запросов.
- `AsyncEngine` в `infrastructure/` инициализируется при старте через lifespan.

### Миграции: Alembic

**Alembic** с асинхронным `env.py` (используя `run_sync` / asyncio-режим).

- Ревизии генерируются из SQLAlchemy-метаданных — синхронизированы с моделями.
- Накатываются одной командой: `alembic upgrade head`.
- Хранятся в `backend/migrations/`.

### Конфигурация: pydantic-settings

**pydantic-settings** читает переменные из `.env`, обязательные поля без дефолтов — ошибка при старте с понятным сообщением. Принцип унаследован от `bot/config.py`.

## Рассмотренные альтернативы

### Django + Django ORM

**Плюсы:** batteries-included, встроенный admin, ORM.  
**Минусы:** избыточность для API-ядра без шаблонов; async-поддержка ORM зрелее в SQLAlchemy 2.x; Django ограничивает структуру каталогов — не совпадает с целевой раскладкой `vision.md` §8.  
**Вывод:** не выбран.

### Синхронный SQLAlchemy (1.x) + psycopg2

**Плюсы:** более знакомый API у части разработчиков.  
**Минусы:** блокирующие вызовы в async ASGI-приложении требуют `run_in_executor` или отдельных потоков — усложнение без выигрыша; SQLAlchemy 2.x несовместим со старым синхронным API в async-контексте без явных мер.  
**Вывод:** не выбран.

### Сырой SQL (asyncpg напрямую)

**Плюсы:** полный контроль, без ORM-абстракций.  
**Минусы:** ручное управление схемами и миграциями; отсутствие единого языка модель↔SQL при реляционной `data-model.md` с ~8 сущностями — выше риск ошибок.  
**Вывод:** допустим для точечных запросов внутри репозиториев, но не как основной подход.

### Tortoise ORM

**Плюсы:** async-first, простой API.  
**Минусы:** менее распространён; Alembic не поддерживает его «из коробки» — сложнее согласованный путь миграций.  
**Вывод:** не выбран.

## Последствия

**Положительные**

- Единый async-стек от HTTP до БД — без блокирующих вызовов в event loop.
- OpenAPI-схема генерируется автоматически → основа контракта для задачи 02 и клиентов.
- Структура каталогов `backend/app/` с `api/`, `domain/`, `infrastructure/`, `services/` соответствует `vision.md` §8.
- Миграции воспроизводимы и управляемы — выполнен DoD итерации 3 (`tasklist-database.md`).

**Отрицательные / риски**

- Async SQLAlchemy требует аккуратного управления временем жизни сессии (в Depends); при неправильном использовании — утечки соединений.
- Alembic `env.py` с asyncio сложнее, чем синхронный вариант; требует явной настройки.
