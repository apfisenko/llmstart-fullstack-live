# Техническое видение проекта

Telegram-бот для сопровождения учебного потока курса AI-driven fullstack developer.

---

## 1. Технологии и инструменты

| Категория | Инструмент | Обоснование |
|---|---|---|
| Язык | Python 3.12+ | стандарт для AI/LLM-проектов |
| Виртуальное окружение | `venv` | встроен в Python, без лишних зависимостей |
| Управление зависимостями | `uv` | быстрый, совместим с `pyproject.toml`, заменяет `pip` + `pip-tools` |
| Telegram Bot API | `aiogram 3.x` + polling | асинхронный, polling — просто и достаточно для MVP |
| LLM | `openai` Python SDK → Openrouter | единый клиент, Openrouter даёт доступ к разным моделям через один API-ключ |
| Автоматизация | `make` | `Makefile` для `install`, `run`, `lint`, `format` |
| Линтер / форматтер | `ruff` | один инструмент вместо `flake8` + `black` + `isort` |
| Конфиг | `pydantic-settings` | валидация типов и чтение `.env` из коробки |
| Зависимости | `pyproject.toml` | единый файл для `uv` |

---

## 2. Принципы разработки

- **KISS** — каждый модуль делает одно и решает одну задачу; никакой «магии» без явной необходимости
- **YAGNI** — фичи появляются по факту требования, не «на вырост»
- **DRY** — повторяющаяся логика выносится в отдельную функцию или класс один раз
- **ООП: 1 класс = 1 файл** — каждый класс живёт в своём файле; имя файла — snake_case от имени класса
- **Async-first** — весь I/O (Telegram API, LLM API) асинхронный через `asyncio` / `aiogram`
- **Fail fast** — при неверной или отсутствующей конфигурации падаем сразу с понятным сообщением
- **Явная конфигурация** — никаких захардкоженных значений; всё через переменные окружения
- **Type hints везде** — проверяется `ruff`; читаемость важнее краткости

## 4. Архитектура проекта

Слоистая архитектура без лишних уровней:

```
Telegram User
     │
     ▼
[ aiogram polling ]
     │
     ▼
[ Handler ]        ← принимает update, формирует ответ, не знает об LLM напрямую
     │
     ▼
[ LlmService ]     ← единственная точка взаимодействия с OpenAI/Openrouter
     │
     ▼
[ Openrouter API ]
```

**Ключевые решения:**
- Нет БД в MVP — состояние живёт только в памяти текущего процесса
- `Handler` импортирует `LlmService`, но не `openai` напрямую
- `Config` загружается один раз при старте и передаётся через dependency injection
- При неверном конфиге — немедленное завершение с понятной ошибкой

## 5. Модель данных

Нет БД в MVP. Все данные — структуры в памяти процесса, сбрасываются при рестарте.

**Сообщение** — стандартный формат OpenAI Chat Completions:
```python
{"role": "user" | "assistant" | "system", "content": str}
```

**История диалога (in-memory):**
```python
history: dict[int, list[dict]] = {}
# int — chat_id из Telegram
# list[dict] — список сообщений в формате выше
```

**Скользящее окно:** хранятся последние 20 сообщений на каждый `chat_id`.

**Конфигурация** (`pydantic-settings`):
```python
telegram_token: str
openrouter_api_key: str
openrouter_model: str        # "openai/gpt-4o-mini"
openrouter_base_url: str     # "https://openrouter.ai/api/v1"
openrouter_timeout: int      # 30 (секунд)
system_prompt_path: str      # "bot/prompts/system.txt"
max_history_size: int        # 20
log_level: str               # "INFO"
```

## 6. Работа с LLM

**Клиент:** `openai` Python SDK с `base_url = https://openrouter.ai/api/v1`.

**`LlmService` отвечает за:**
1. Хранение истории диалогов (`dict[chat_id → list[messages]]`)
2. Добавление системного промпта первым элементом при каждом запросе (не сохраняется в историю)
3. Обрезку истории до последних `MAX_HISTORY_SIZE` сообщений
4. Вызов `client.chat.completions.create(...)` и возврат текста ответа
5. Добавление ответа ассистента в историю

**Системный промпт:** читается из `bot/prompts/system.txt` один раз при старте.

**Модель по умолчанию:** `openai/gpt-4o-mini`.

**Таймаут:** 30 секунд, берётся из конфига.

**Обработка ошибок:** при сбое LLM API бот отвечает пользователю понятным сообщением и логирует исключение; история не обновляется; бот не падает.

## 7. Сценарии работы

**Сценарий 1: Обычный вопрос по курсу**
```
Пользователь → пишет текстовое сообщение
Bot → отправляет статус "печатает..." (send_chat_action)
Bot → добавляет сообщение в историю chat_id
Bot → отправляет [system prompt + история] в LLM
LLM → возвращает ответ
Bot → добавляет ответ в историю, отправляет пользователю
```

**Сценарий 2: Ошибка LLM API**
```
Пользователь → пишет текстовое сообщение
Bot → запрос к LLM упал (таймаут, сетевая ошибка и т.п.)
Bot → логирует ошибку (ERROR)
Bot → отвечает: "Не удалось получить ответ. Попробуйте позже."
История → не обновляется
```

**Сценарий 3: Нетекстовое сообщение**
```
Пользователь → отправляет фото, стикер, файл и т.п.
Bot → отвечает: "Работаю только с текстовыми сообщениями."
История → не обновляется
```

**Команды бота:**
- `/start` — приветствие и краткое описание возможностей; история не сбрасывается
- `/help` — аналогично `/start`
- `/reset` — сброс истории диалога для текущего `chat_id`

## 8. Подход к конфигурированию

- Все настройки — через переменные окружения, файл `.env` (не коммитится)
- `.env.example` коммитится в репозиторий — шаблон со всеми ключами без значений
- Загрузка и валидация при старте — один класс `Config` в `bot/config.py` на базе `pydantic-settings`
- Если обязательная переменная отсутствует — падаем сразу с понятным сообщением (fail fast)
- Нет вложенных конфигов, нет YAML/TOML для runtime — только `.env`

**Шаблон `.env.example`:**
```dotenv
TELEGRAM_TOKEN=
OPENROUTER_API_KEY=
OPENROUTER_MODEL=openai/gpt-4o-mini
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
OPENROUTER_TIMEOUT=30
SYSTEM_PROMPT_PATH=bot/prompts/system.txt
MAX_HISTORY_SIZE=20
LOG_LEVEL=INFO
```

## 9. Подход к логированию

- Стандартный модуль `logging` из stdlib — никаких дополнительных зависимостей
- Настройка один раз в `bot/utils/logger.py`; в каждом модуле — `logging.getLogger(__name__)`
- Формат: `%(asctime)s | %(levelname)s | %(name)s | %(message)s`
- Уровень логирования берётся из конфига (`LOG_LEVEL`)
- Вывод только в `stdout` — без ротации файлов

**Что логируем:**

| Уровень | Событие |
|---|---|
| `INFO` | Старт бота, получено сообщение (хэш `chat_id`, тип), отправлен ответ |
| `WARNING` | Нетекстовое сообщение, сброс истории (`/reset`) |
| `ERROR` | Ошибка LLM API, необработанное исключение |
| `DEBUG` | Полный контент запроса/ответа LLM (только при `LOG_LEVEL=DEBUG`) |

`chat_id` в логах — только в виде хэша (приватность пользователей).
