# DevOps Tasklist

## Обзор

Подготовительная часть DevOps: локальный полный стек в Docker, единый `docker-compose`, удобные команды и документация; затем сборка и публикация образов в **GitHub Container Registry (GHCR)** через GitHub Actions. Далее — **VPS в Timeweb Cloud** (официальный CLI **`twc`**, [документация RU](https://raw.githubusercontent.com/timeweb-cloud/twc/refs/heads/master/docs/ru/README.md)), **минимальная достаточная** конфигурация сервера с **публичным внешним IP**; настройка хоста, ручной запуск стека с образов GHCR, затем **автоматический деплой** при изменении кода в репозитории.

**Политика секретов:** не коммитить; в репо — только [`.env.ghcr.example`](../../.env.ghcr.example) и иные `*.example`, плюс отсылки к **GitHub Secrets** и файлам на сервере вне git. Для SSH — **два** ключа: **персональный** (разработка/отладка) и **для CI/CD** (автодеплой). **`docker login ghcr.io` на сервере** — только **ручной** шаг: в документации — команды и инструкция; токен и ввод — выполняет пользователь (не агент).

## Итерация 1: локальный полный стек {#iteration-1-local-stack}

- [x] Зафиксировать схему для **PostgreSQL**: init в [`devops/postgres/`](../../devops/postgres/); каталог [`docker/postgres/`](../../docker/postgres/) — редирект в README без дублирования SQL.
- [x] Добавить **Dockerfile** для backend, bot, web (Next.js multi-stage; Python — uv; секреты не в образах). **`.dockerignore`** — один в корне репозитория (см. `devops/README.md`).
- [x] Расширить **корневой** `docker-compose.yml`: полный стек в **одном** файле; сервис `postgres` — единый источник правды для режима «только БД» и для полного стека.
- [x] Обеспечить **отдельный запуск только БД** без приложений: профиль **`app`** для приложений; `postgres` без профиля; `db-up` / `db-down` — только Postgres + миграции с хоста по документации.
- [x] Расширить [tasks.ps1](../../tasks.ps1) и [Makefile](../../Makefile): `stack-up` / `stack-down`, `stack-status`, `stack-logs`, `check-backend`, `check-web`, `check-bot` (согласованные имена).

### Definition of Done

**Для агента (self-check)**

- `devops/README.md` объясняет структуру и отсутствие дублирования compose-файлов «только БД» при отдельной процедуре `db-up`.
- `docker compose config` проходит без ошибок; `docker compose up` поднимает согласованный набор сервисов; **только postgres** поднимается отдельной документированной командой / профилями без ручного редактирования YAML «на лету».
- Существуют Dockerfile и `.dockerignore` для backend, bot, web; образы собираются из корня репозитория ожидаемыми путями.
- `tasks.ps1` и `Makefile` содержат согласованные цели: отдельно БД и отдельно полный стек + диагностика (`status`, `logs`) и **отдельные команды проверки для backend, frontend и bot** (ожидаемый успешный выход задокументирован).

- Создан файл инструкции в `docs/tech/`; обновлены `architecture.md`, `onboarding.md`, при необходимости README и `plan.md`.

**Для пользователя (user-check)**

- По инструкции можно поднять **только** PostgreSQL и подключить к нему backend на хосте (как раньше с `db-up`).
- По инструкции можно поднять **полный стек** и по отдельности убедиться, что **backend**, **frontend** и **bot** отвечают ожидаемо (команды из tasklist/инструкции).
- Команды в `.\tasks.ps1 help` и `make`/`make help` понятны без чтения исходников compose.

## Итерация 2: образы в GHCR и локальная проверка {#iteration-2-ghcr}

### Цель

Образы backend, bot и web **собираются в CI** и публикуются в **GHCR**; локально можно поднять проект **с образов из registry**, без обязательной локальной `docker build` для ежедневной проверки.

### Состав работ

- [x] Добавить workflow GitHub Actions по skill **`github-actions-templates`**: job(ы) с `docker build` и `docker push` в `ghcr.io/<owner>/<repo>/...`; permissions `packages: write`; теги минимум **`latest`** для default branch и **`sha`** (или semver при релизах — зафиксировать в workflow/README пакета).
- [x] Документировать имена образов и теги (в `devops/README.md`, инструкции или `docs/tech/`).
- [x] Подготовить сценарий Compose **из опубликованных образов** (override-файл, отдельный compose-фрагмент или `profiles` — один понятный способ «как на registry») и провести **ревью** этой конфигурации (в т.ч. через **docker-expert** при существенных отличиях от локального build).
- [x] Проверить: полный подъём стека **только из образов GHCR** (переменные/registry login при необходимости задокументировать).

### Definition of Done

**Для агента (self-check)**

- Workflow успешно выполняется на целевой ветке; в GHCR видны ожидаемые репозитории образов и теги.
- Документация содержит точные имена `image:` и способ логина к GHCR для `docker pull` при приватном пакете.
- `docker compose` с образами из registry поднимает сервисы; миграции/порядок старта согласованы с итерацией 1.
- Ревью compose «из registry» выполнено, замечания учтены.

**Для пользователя (user-check)**

- После успешного CI на основной ветке образы появляются в GHCR и их можно найти по инструкции.
- Склонировав репозиторий и задав теги env (если нужно), можно поднять стек **без** локальной сборки приложений.

### Артефакты (целевые)

- `.github/workflows/*.yml` — сборка и публикация в GHCR.
- Документ(ы) с тегами и примером `compose` из registry.
- При необходимости — `docker-compose.override.yml` / отдельный yaml для registry (как принято в репозитории после итерации 1).

Получить **воспроизводимо** минимальную по стоимости (но достаточную под стек проекта) VPS с **публичным внешним IP** в **Timeweb Cloud**, доступ по **SSH с ключом**; зафиксировать в документации **идентификацию** сервера без секретов.

- [x] **Проверка:** вход по SSH с **персональным** ключом успешен; **ключ CI/CD** доведён до состояния «можно зайти с машины, где лежит только deploy-ключ» (проверка на шаге ит. 4–5, если нет отдельного runner). — чеклист в [timeweb-vps.md](../tech/timeweb-vps.md); фактическая проверка — владелец на своей машине / в ит. 4–5

**Для пользователя (user-check)**

## Итерация 4: настройка сервера и ручной деплой {#iteration-4-server-setup}

### Цель

Сервер готов к запуску **того же** стека, что описан для образов GHCR: [docker-compose.ghcr.yml](../../docker-compose.ghcr.yml), [docs/tech/docker-compose-ghcr.md](../tech/docker-compose-ghcr.md). Установлены **Docker** и **Docker Compose plugin**, утилиты **обслуживания** (например просмотр логов, `curl` для health); подготовлены **каталог** деплоя и **`.env` на сервере** по образцу; выполнен **ручной** `docker login ghcr.io` и **ручной** подъём `docker compose`; подтверждена работа **API** и **веб** в браузере.

### Состав работ

- [x] Установить и задокументировать всё **необходимое** для работы приложения и **удобного** сопровождения (список команд/пакетов — в `docs/tech/`).
- [x] **Повторяемый** способ: **скрипт** в репозитории (одно согласованное место, напр. `devops/vps/` или `scripts/vps/`) **и** пошаговая инструкция в `docs/tech/`; итог — другой человек или агент может **повторить на новом сервере** (порядок «сначала скрипт, потом дока» — на выбор исполнителя).
- [x] Подготовить на сервере **каталог** деплоя (клон репо или копия compose + env) и **`.env` только по образцу** [`.env.ghcr.example`](../../.env.ghcr.example); инструкция: **что** скопировать из репо, **что** задать только на сервере (секреты — в файлах/переменных **вне** git или в GitHub Secrets для CD).

- [x] **Ручной** запуск стека: `docker compose` с [docker-compose.ghcr.yml](../../docker-compose.ghcr.yml) (или согласованный compose-вход) и env на сервере.

## Итерация 5: автоматизация деплоя (GitHub Actions) {#iteration-5-cd-gha}

### Цель

**Автоматический** деплой на сервер при **изменении кода** в репозитории (триггер: **`push`** на согласованную ветку вроде `main` / `master` — зафиксировать в workflow; при необходимости `workflow_call` от job GHCR). Используются **github-actions-templates**-практики: `permissions`, отсутствие утечек секретов в логи.

### Состав работ

- [ ] Документация (в `docs/tech/` или отдельный раздел): какие **GitHub Secrets** создать (например **хост** или `DEPLOY_HOST`, **пользователь** SSH, **приватный ключ CI/CD** в `-----BEGIN`…`END-----`, при необходимости порт; **фingerprint** `known_hosts` — по согласованию). Указать: секреты создаёт **владелец** репозитория/организации; **не** дублировать значения в markdown.
- [ ] Настроить **workflow** в [`.github/workflows/`](../../.github/workflows/): деплой **после** успешной публикации образов или **в** том же pipeline после зелёного `docker push` (связь с [ghcr.yml](../../.github/workflows/ghcr.yml) — `needs`, `workflow_run`, или `workflow_call` — выбрать и описать). Типовая схема: по SSH на сервер — `docker compose pull` и `up -d` (или `git pull` + compose — **один** согласованный способ) в каталоге деплоя.
- [ ] Следовать skill [**`github-actions-templates`**](../../.cursor/skills/github-actions-templates/SKILL.md): `permissions`, `concurrency` по желанию, **никогда** не `echo` секреты.
- [ ] **Проверка:** осмысленное изменение в коде/конфиге (пуш), ожидание **зелёного** workflow, затем **та же** проверка, что в итерации 4 (API + браузер) без ручного деплоя.
- [ ] Обновить документацию: [docs/tech/docker-compose-ghcr.md](../tech/docker-compose-ghcr.md) — раздел **CD**; [docs/plan.md](../plan.md) — критерий **«воспроизводимый production-deploy»** / итерация 6; [architecture.md](../architecture.md) при необходимости.

### Definition of Done

**Для агента (self-check)**

- Workflow деплоя выполняется на целевой ветке; job **успешен**; в логах **нет** значений Secrets.
- Документация перечисляет имена Secrets и **назначение** каждого; путь/команда на сервере однозначны.
- Повторный пуш/деплой **идемпотентен** в обычном смысле (`compose up` не ломает стабильный стек).

**Для пользователя (user-check)**

- Могу по docs создать Secrets, **не** кладя ключи в репо, и убедиться, что после пуша образы **обновляются** на сервере/стек перезапускается.
- Health и веб открываются **после** автодеплоя так же, как после ручного шага (ит. 4).

### Артефакты (целевые)

- `.github/workflows/<deploy>.yml` (или расширение существующего) — CD.
- Раздел в [docs/tech/docker-compose-ghcr.md](../tech/docker-compose-ghcr.md) или отдельный `docs/tech/…` с **списком Secrets** и сценарием ручного отката (по желанию — кратко).
- Обновления [plan.md](../plan.md), [architecture.md](../architecture.md).

## Вне scope (граница этого tasklist)

**В scope** итераций 3–5: VPS **Timeweb Cloud** и **`twc`**, настройка сервера, ручной `docker login` + ручной подъём, затем **автодеплой через GitHub Actions** при изменении кода.

**Вне scope** (последующие этапы, если не взяты отдельно): Kubernetes/Helm, **полноценный** мониторинг и алерты, централизованные бэкапы БД в облаке, мульти-регион, смена провайдера VPS, **observability** в смысле [vision.md](../vision.md) «прод-наблюдаемость» сверх логов контейнеров, **управление прод-секретами** вне схемы «`.env` на сервере + GitHub Secrets».
