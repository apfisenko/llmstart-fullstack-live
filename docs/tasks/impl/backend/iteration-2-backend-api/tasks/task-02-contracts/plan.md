# Задача 02: Проектирование API-контрактов

## Ресурсы и операции

Все пути относительно базового URL backend (например `http://127.0.0.1:8000`).

| Метод | Путь | Назначение |
|-------|------|------------|
| `POST` | `/api/v1/cohorts/{cohort_id}/dialogues/messages` | Отправить сообщение: создать/продолжить `Dialogue`, сохранить `DialogueMessage`, вызвать LLM, вернуть ответ ассистента |
| `POST` | `/api/v1/dialogues/{dialogue_id}/reset` | Сброс контекста диалога (аналог `/reset` в боте) |
| `GET` | `/api/v1/cohorts/{cohort_id}/progress-checkpoints` | Список этапов `ProgressCheckpoint` для потока |
| `PUT` | `/api/v1/cohorts/{cohort_id}/memberships/{membership_id}/progress-records/{checkpoint_id}` | Upsert `ProgressRecord` по паре (участие, этап) |
| `GET` | `/api/v1/cohorts/{cohort_id}/summary` | Агрегированный срез по потоку для преподавателя (участники + прогресс по этапам) |

Служебный health и документация Swagger — вне этого контракта v1: см. задачу 03 (`GET /health`, `/docs` при включении).

## Сопоставление с data-model

| Сущность | Где в API |
|----------|-----------|
| `Cohort` | `cohort_id` в пути |
| `CohortMembership` | `membership_id` в теле (`POST ... dialogues/messages`) и в пути (progress-records) |
| `Dialogue` | создаётся/продолжается при `POST ... dialogues/messages`; `dialogue_id` в ответе и опционально в запросе |
| `DialogueMessage` | `user_message_id`, блок `assistant_message.{id, content, created_at}` в ответе |
| `ProgressCheckpoint` | элементы списка в `GET ... progress-checkpoints` и в `CohortSummaryResponse.checkpoints` |
| `ProgressRecord` | `PUT ... progress-records/{checkpoint_id}` и поля в summary |

Имена полей в JSON — **snake_case**, UUID — строки в формате UUID.

## Статусы прогресса (enum API)

Согласованы с [docs/data-model.md](../../../../../../data-model.md): `not_started`, `in_progress`, `completed`, `skipped`.

## Идентификация и доступ (MVP)

- **Схема в OpenAPI:** `bearerAuth` (заголовок `Authorization: Bearer <token>`).
- **Смысл токена на MVP:** общий секрет клиента (бот/веб) из окружения **или** заглушка под будущий JWT. Конкретное значение и ротация — в `.env` / задаче 05, не в этом файле.
- **Авторизация по сущностям:** сервер проверяет, что `membership_id` / `dialogue_id` принадлежат тому же `cohort_id` и вызывающему субъекту (правила ролей: студент — свой membership; преподаватель — обзор потока по `GET ... summary`).
- Ответы: **401** нет/неверный токен; **403** токен есть, но операция запрещена для роли/ресурса.

## Ошибки и коды HTTP

Единое тело (предпочтительно):

```json
{
  "error": {
    "code": "SOME_CODE",
    "message": "Человекочитаемое описание",
    "details": null
  }
}
```

| Код | Когда |
|-----|--------|
| 400 | Семантически неверный запрос (редко, если отделён от 422) |
| 401 | Нет аутентификации |
| 403 | Нет прав |
| 404 | Ресурс не найден |
| 409 | Конфликт домена |
| 422 | Тело не проходит схему (стандарт FastAPI/Pydantic для валидации) |
| 502 | Ошибка/некорректный ответ провайдера LLM (`LLM_BAD_GATEWAY`) |
| 503 | Провайдер LLM недоступен (`LLM_UNAVAILABLE`) |

**Задача 05:** при необходимости добавить **exception handler**, чтобы привести ответы FastAPI (в т.ч. `422` с полем `detail`) к единому виду `error` — или явно задокументировать два формата на переходный период.

Тексты ошибок для клиента **не** должны содержать содержимое переписки или внутренние трейсы ([vision.md](../../../../../../vision.md) §15).
