### Уточнения по зоне A

- **KPI (4):** число активных студентов в потоке; число завершённых домашних чекпоинтов (по `ProgressRecord` с `completed` и `checkpoint.is_homework`); средний процент прогресса по группе; число вопросов к ассистенту (ходов `DialogueTurn` в каналах `telegram` и/или `web` — в контракте явно: «все каналы потока») за текущую календарную неделю vs предыдущая неделя.
- **Лента вопросов:** пагинация cursor + опциональный поиск `q` по тексту вопроса (серверный фильтр); ответы в теле элементов ленты (преподавательский контур).
- **Лента сдач:** последние N записей прогресса со статусом `completed` (и при необходимости `in_progress` с комментарием — по согласованию в задаче 02).

### Уточнения по зоне B

- Таблица и scatter используют один ответ лидерборда: клиент не обязан дублировать расчёт рейтинга.
- Оси scatter: горизонталь — доля или число завершённых обычных чекпоинтов (не ДЗ); вертикаль — завершённые ДЗ (`is_homework`).

### Уточнения по зоне C

- «Заблокировано» для будущих уроков — правило отображения на клиенте по `sort_order` и текущем статусе (детали последовательности в задаче 03); сервер отдаёт факты статусов.
- Ссылки при сдаче — массив строк в теле `PUT progress-records` (`submission_links`), не смешивать с произвольным `comment`.

## Решение по архитектуре API (зона A)

Для MVP выбран **один агрегированный ответ** `GET .../teacher-dashboard`: меньше round-trip для Next.js, проще загрузить экран как на макете [`dashboard.png`](../../../../../ui/dashboard.png). Расширение `GET .../summary` не делается в этой итерации контракта, чтобы не смешивать старый контракт преподавательского среза с тяжёлым дашбордом; при необходимости задача 02 может дублировать часть данных между handler'ами.

## Вход по Telegram username (MVP)

1. **Продукт:** пользователь вводит username с `@` или без; нормализация на сервере (trim, без `@`, регистр по соглашению в задаче 02).
2. **Домен:** **принято решение** — персистентное поле **`telegram_username`** у сущности `User` / колонка в **`users`** (уникальность среди ненулевых, нормализация в приложении). Вариант «только маппинг в сиде без поля в БД» для контракта `username → user` **не выбирается**; сид в задаче 02 заполняет ту же колонку. Подробнее — в [`docs/data-model.md`](../../../../../data-model.md) §`User`; миграция и ORM — **задача 02**.
3. **HTTP:** **`POST /api/v1/auth/dev-session`** с телом `{ "telegram_username": "string" }`, заголовок `Authorization: Bearer` как у остальных v1. Ответ: `user_id`, `display_name`, список `memberships[]` (для MVP достаточно одного активного потока в сиде), опционально `web_dialogue_id` если сервер создаёт/находит активный диалог канала `web` сразу. **404** — пользователь не найден. Это не JWT: контекст выбранного участия веб хранит у себя и подставляет в `membership_id`, `viewer_membership_id`, пути с `cohort_id`.

Query-параметр `telegram_username` для `GET` не используется (username в URL/query попадает в логи прокси).

## Навигация веб → вызовы API (после dev-session)

| Маршрут / экран | Вызовы |
|-----------------|--------|
| Вход | `POST /api/v1/auth/dev-session` |
| Панель преподавателя | `GET .../teacher-dashboard?viewer_membership_id=...` |
| Лидерборд | `GET .../leaderboard?viewer_membership_id=...` (или без viewer для публичного режима — политика в задаче 02), `GET .../progress-checkpoints` |
| Кабинет студента | `GET .../progress-overview`, `PUT .../progress-records/{checkpoint_id}` |
| Чат (FAB) | `GET .../dialogues/{dialogue_id}/turns`, `POST .../dialogues/messages`, `POST .../dialogues/{id}/reset` |

## Передача в задачу 02 (backend)

Реализовать в коде и БД (без исчерпывающего повторения OpenAPI здесь):

1. **Миграции / ORM:** колонка `telegram_username` у пользователя; колонка или JSON для **`submission_links`** у `ProgressRecord` (предпочтительно `jsonb` массив строк URL либо `text[]`); при необходимости индексы под ленты (`dialogue_turns.asked_at` + join на поток уже описаны в data-model).
2. **Эндпоинты:** `POST /api/v1/auth/dev-session`; `GET /api/v1/cohorts/{cohort_id}/teacher-dashboard`; `GET /api/v1/cohorts/{cohort_id}/leaderboard`; `GET /api/v1/cohorts/{cohort_id}/memberships/{membership_id}/progress-overview`; `GET /api/v1/dialogues/{dialogue_id}/turns`.
3. **Права:** те же правила, что для существующих маршрутов — Bearer, проверка `membership_id`/`dialogue_id`/`cohort_id` и ролей; лидерборд — уточнить публичность (студент видит только свой поток и т.д.).
4. **Сиды:** mock-данные и пользователь преподавателя `@akozhin` (как в tasklist-02), заполнение `telegram_username`.
5. **Синхронизация:** `ProgressCheckpointItem` в ответах API с полем `is_homework`; выравнивание расхождений ORM/БД из §«Соответствие реализации» в [`data-model.md`](../../../../../data-model.md) по мере необходимости.

Гостевые маршруты `POST /api/v1/assistant/guest/*` остаются вне обязательного объёма задачи 02 для веб-MVP, если уже есть в коде — не ломать; в OpenAPI приведены к одному стилю при наличии.
