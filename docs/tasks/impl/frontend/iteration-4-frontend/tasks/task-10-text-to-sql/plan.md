# Задача 10: ответы на вопросы по данным БД (Text-to-SQL) — план

**Статус:** ⛔ **Не выполняется** (после анализа плана — усложнение; оставлено на дальнейшее развитие). Реализация не ведётся.  
**Связь:** [`docs/tasks/tasklist-frontend.md`](../../../../../../tasks/tasklist-frontend.md) (блок 5), [`docs/tech/api-contracts.md`](../../../../../../tech/api-contracts.md), [`docs/tech/frontend-requirements.md`](../../../../../../tech/frontend-requirements.md) §9.

## 1. Цель

Дать ассистенту возможность отвечать на вопросы пользователя, **опираясь на фактические данные PostgreSQL** (агрегаты по потоку, прогресс, диалоги и т.д. — в рамках политики доступа), с **безопасной** архитектурой (без произвольного destructive SQL и утечек схемы/данных в логи).

**Паритет каналов:** поведение распространяется на **веб и Telegram-бот** за счёт реализации **только в `backend/`** на общем пути обработки сообщения диалога (`POST /api/v1/cohorts/{cohort_id}/dialogues/messages` для `channel: web` и `channel: telegram`). Бот не реализует Text-to-SQL и не ходит в БД; при необходимости — только передача уже согласованных полей тела запроса и UX-тексты (см. §6).

## 2. Ограничения и принципы

- Ядро — backend ([`docs/vision.md`](../../../../../../vision.md)); контракт один на клиентов ([`docs/tech/api-contracts.md`](../../../../../../tech/api-contracts.md) §«Один контракт для каналов»).
- Не логировать тексты переписки, сгенерированный SQL и результаты выборок в stdout ([`vision.md`](../../../../../../vision.md) §15; правила `.cursor/rules/conventions.mdc`).
- Секреты и политика — только из окружения (отдельный DB-пользователь read-only, allowlist, лимиты — см. §4).
- Гостевой контур `POST /api/v1/assistant/guest/*` **вне MVP задачи 10**, если не принято явное расширение: там нет персистентного домена потока — включение Text-to-SQL там требует отдельного ADR.

## 3. Варианты архитектуры (выбор до кодирования)

Зафиксировать выбранный вариант в **`docs/adr/`** (предпочтительно для безопасности и аудита) или кратко в `docs/tech/text-to-sql.md`, если объём маленький.

| Вариант | Плюсы | Минусы / риски |
|--------|--------|------------------|
| **A. Tool-calling / двухшаговый pipeline** (LLM предлагает SQL → валидатор → выполнение read-only) | Гибкие формулировки вопросов; контроль на втором шаге | Нужны жёсткий валидатор, allowlist, лимиты строк/времени |
| **B. Ограниченный «каталог» запросов** (шаблоны / заранее разрешённые id отчётов) | Максимальная безопасность | Слабая гибкость «произвольного» вопроса |
| **C. RAG по схеме + генерация SQL** | Лучше соответствие схеме | Всё равно нужен исполнитель с политикой как в A |
| **D. Только NL→API** (без SQL: маппинг вопроса на существующие агрегатные endpoint'ы) | Нет произвольного SQL | Ограниченный класс вопросов |

**Рекомендация для первой поставки:** **A** (или **D** как промежуточный шаг, если срок критичен) с обязательными мерами из §4. Комбинация **D→A** допустима: сначала покрыть 80% вопросов безопасными вызовами доменных сервисов, затем узкий Text-to-SQL на allowlist-таблицах.

## 4. Безопасность исполнения (обязательный минимум)

- Отдельная роль PostgreSQL **только `SELECT`**, без прав на DDL/DML, на ограниченном наборе схем/таблиц.
- **Allowlist** имён таблиц/представлений (или только `public` views, спроектированных под отчёты).
- Валидация AST / запрет нескольких statement, запрет `;`, функций, `COPY`, `INTO`, комментариев с обходом — по согласованному уровню параноидальности.
- Жёсткие лимиты: **timeout** запроса, **max rows**, **max bytes** ответа; при превышении — контролируемая ошибка в pipeline (пользователю — обобщённое сообщение, без SQL).
- Опционально: второй LLM-pass «ревью SQL» — только если оправдано бюджетом latency/стоимости.

## 5. Место внедрения в коде (`backend/`)

- Точка расширения: сервис диалога, вызываемый из [`routes_dialogues.py`](../../../../../../../backend/app/api/v1/routes_dialogues.py) (`DialogueService.post_message` / аналог), и/или слой вызова LLM [`llm_assistant`](../../../../../../../backend/app/infrastructure/llm_assistant.py) — **после** проверок membership/cohort/channel.
- В контекст модели передавать: роль пользователя, `cohort_id`, `membership_id` (и правила, какие сущности видны студенту vs преподавателю) — **не** передавать в клиент сырой SQL и сырые дампы.
- Идемпотентность и сохранение ходов диалога — без изменения контракта ответа `assistant_message` (расширения JSON — только необязательные поля в рамках v1, если нужно «источник: данные» для UI).

## 6. Веб (`frontend/web/`)

- По tasklist: **UI опционален** — достаточно того, что чат уже бьёт в тот же `POST .../dialogues/messages`.
- Если нужен явный режим «вопрос к данным»: либо **флаг в теле** запроса (тогда обновить OpenAPI + типы клиента + тело в боте), либо **политика на сервере** (фича-флаг окружения / роль) — тогда **изменений в боте нет**.
- Любой новый флаг — описать в [`api-contracts.md`](../../../../../../tech/api-contracts.md) и [`openapi-v1.yaml`](../../../../../../api/openapi-v1.yaml).

## 7. Бот (`bot/`)

- Текущий путь: [`BackendAssistantService._chat_cohort`](../../../../../../../bot/services/backend_assistant.py) — `POST .../dialogues/messages` с `channel: "telegram"`.
- **Целевое состояние:** при включённой фиче на backend ответы в Telegram **автоматически** используют тот же data-grounding pipeline, что и веб, без дублирования логики.
- Изменения в боте — **только при** введении обязательного поля в теле запроса или отдельного endpoint'а; иначе ограничиться документацией ([`integrations.md`](../../../../../../integrations.md)) и приёмочными сценариями для бота.
- Логирование в боте: по существующим правилам (длины, хэш `chat_id`), без текста вопроса/ответа.

## 9. Тест-сценарии (черновик для `summary` / `backend/tests/`)

**Позитивные (cohort-диалог, web и telegram — один набор логики на сервере):**

- Вопрос с агрегатом по потоку, ответ согласован с фикстурой БД.
- Вопрос студента — не видит чужие персональные данные вне своего membership.

**Негативные / безопасность:**

- Попытка DDL/DML/мульти-стейтмент — отклонение валидатором, пользователю без деталей.
- Запрос к таблице вне allowlist — отказ.
- Превышение лимита строк/времени — контролируемый отказ.

**Регрессия:** обычный «общий» вопрос к ассистенту без обращения к БД остаётся рабочим.

## 10. Критерии готовности (согласование с tasklist)

- Записано архитектурное решение (ADR или tech-док).
- Реализация в backend с политикой безопасности; нет опасных запросов вне allowlist/read-only.
- Веб и бот: либо **нулевые** изменения в боте (серверная политика), либо синхронизированные изменения тела запроса + OpenAPI + оба клиента.
- Тесты и/или сценарии из §9 отражены в `backend/tests/` или в `summary.md` задачи.

## 11. Риски и зависимости

- Пересечение с roadmap итераций **5–6** (интеграция клиентов, инфраструктура): отдельный read-only пользователь БД и миграции прав — возможна задача инфраструктуры.
- Стоимость и задержка LLM при двухшаговом pipeline; нужны лимиты и деградация («не могу ответить по данным» без утечки внутренностей).
