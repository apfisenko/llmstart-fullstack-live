# Задача 01: сценарии студент/преподаватель и требования к данным

## Контекст

- **Продукт:** [`docs/idea.md`](../../../../../../idea.md) — сопровождение потока, фиксация прогресса, бот как первый клиент, веб как целевая поверхность.
- **Архитектура:** [`docs/vision.md`](../../../../../../vision.md) — ядро backend, роли студент/преподаватель, сценарии на уровне vision нужно сузить под **учебный этап** и будущую БД.
- **Контракт API:** [`docs/tech/api-contracts.md`](../../../../../../tech/api-contracts.md) — имена ресурсов (`Cohort`, `CohortMembership`, `Dialogue`, `ProgressCheckpoint`, `ProgressRecord` и т.д.); требования к данным должны **согласовываться** с этим словарём, без обязательного дублирования путей HTTP в пользовательском тексте.
- **Текущая модель:** [`docs/data-model.md`](../../../../../../data-model.md) — опорная терминология до упрощений задачи 02; в задаче 01 явно не углубляться в типы SQL/ORM.

Источник состава работ и DoD: [`docs/tasks/tasklist-database.md`](../../../../../../tasks/tasklist-database.md), блок 1.

## Цель

Зафиксировать в доступном виде, **что делают** студент и преподаватель (без стека и SQL), и вывести из этого **какие данные** системе нужны и зачем — как вход для задачи 02 (схема) и последующей реализации.

## Объём и ограничения

- Язык описания: **пользовательский** (экраны/действия), не эндпоинты и не миграции.
- Допущения учебного этапа (обязательно явно в артефактах):
  - **`модуль == урок`** — один уровень программы для отображения и прогресса.
  - **`история диалога == пара вопрос + ответ`** (один шаг обмена с ассистентом как минимальная единица для аналитики преподавателя, если сценарий это требует).
- Не расширять продукт сверх tasklist: фокус на сценариях из блока 1 (прогресс/домашки/лидерборд, состав потока, активность по диалогам, агрегаты).

## План работ (порядок выполнения)

1. **Студент — сценарии** (отдельные подразделы в `user-scenarios.md`):
   - просмотр **текущего состояния** собственного прогресса по потоку (что сделано / в работе / дальше);
   - **лидерборд** (или сопоставимое сравнение) по выполнению домашних заданий **среди участников потока** — что пользователь видит и в какой ситуации (без алгоритма ранжирования в технике, но с явным смыслом «кто сколько сделал»).
2. **Преподаватель — сценарии:**
   - список **участников** потока и базовая идентификация роли/статуса участия;
   - **прогресс группы** (сводка по урокам/этапам для потока);
   - **активность за период** в разрезе **диалогов** (вопросы студентов к ассистенту): что значит «период», что считается активностью;
   - **статистика / агрегаты** по истории диалогов (примеры: число вопросов за период, по пользователям, по урокам — на уровне «какие отчёты нужны», не SQL).
3. После каждого сценария — блок **«Нужные данные»**: сущности и связи **словами предметной области** (пользователь, поток, участие, урок, прогресс, диалог, пары сообщений и т.д.), плюс **какие запросы к данным** подразумеваются (например: «все отметки прогресса по потоку», «сообщения пользователя за интервал») — без полей БД.
4. Зафиксировать все допущения учебного этапа в начале или в конце `user-scenarios.md` отдельным разделом.
5. Создать файл **`docs/tech/user-scenarios.md`** и свести туда пункты 1–4.
6. Обновить **`docs/vision.md`** — раздел **«Роли и пользовательские сценарии»** (§4): согласовать формулировки с `user-scenarios.md`, не противореча [`idea.md`](../../../../../../idea.md); сохранить принцип «сквозные сценарии, не привязаны к одному клиенту».
7. **Самопроверка по DoD** из `tasklist-database.md` (чеклист ниже).

## Согласование с API-контрактом

При именовании сущностей в требованиях к данным опираться на [`api-contracts.md`](../../../../../../tech/api-contracts.md) и [`data-model.md`](../../../../../../data-model.md): где возможно использовать те же термины (`Cohort`, `membership`, чекпоинты прогресса, диалог/сообщения). Если сценарий требует понятия, которого нет в контракте — описать как **пробел** для задачи 02 (не править OpenAPI в рамках задачи 01 без отдельного решения).

## Риски и открытые вопросы

- **Лидерборд:** уточнить в документе, достаточно ли агрегата «число выполненных ДЗ/уроков» или нужны веса/дедлайны — зафиксировать минимально достаточное для учебного этапа.
- **Приватность:** лидерборд и сводки по диалогам — что видит студент vs преподаватель; кратко обозначить в сценариях (без юридического текста).
- **Гостевой ассистент** (`/assistant/guest/*`) — вне персистентной БД; в user-scenarios не смешивать с аналитикой потока, либо явно пометить «вне объёма сценариев потока».
